## Questions for Specific Reviewer Types

If you prefer a role-specific question set, use one of these instead of the full form.

### For Technical Reviewers (Developers, CS Students, Engineers)

1. Architecture: Does the mock-provider / provider-adapter pattern make sense to you?
2. Code quality: From the README and docs, does the project appear well-organized?
3. Testing: 97/97 Flutter tests — does this feel sufficient for a portfolio MVP?
4. Documentation: Is the documentation sufficient to onboard a new contributor?
5. Biggest technical gap: What would you add or change first?

### For Non-Technical Users (Friends, Family, Travelers)

1. Do you understand what the app does without reading code?
2. Would you want to use this on a trip? Why or why not?
3. What one feature matters most to you?
4. Was anything frustrating or confusing?
5. Is there anything you expected to see that was missing?

### For Recruiters / Interviewers / Hiring Managers

1. Does this project demonstrate initiative and product thinking beyond a tutorial?
2. Is the documentation sufficient to evaluate the candidate's technical communication?
3. Does the honest disclosure of limitations (mock-only, not production-ready) signal maturity or lack of confidence?
4. Does this project differentiate the candidate from their peers?
5. Would you move this candidate to the next interview stage based on this project alone? Why or why not?

## Final Question (All Versions)

> **Would this project make you more interested in interviewing or collaborating with the builder? Why or why not?**

This is the single most valuable question on this form. Please be brutally honest — the builder is looking to improve, not to be flattered.
